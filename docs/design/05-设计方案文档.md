## 5.1 设计方向总述

### 5.1.1 视觉定位

Hyper-Claw 的视觉定位为**「专业工具感 × 科技温度」**——面向开发者和技术用户，既要传达可靠、高效、专业的工具属性，又不能冰冷僵硬。

**关键词**：克制、沉浸、高信息密度、低视觉噪声

| 维度 | 设计策略 | 对标参考 |
|------|----------|----------|
| 整体调性 | 深色优先，蓝色科技感 | VS Code、GitHub Dark |
| 信息层级 | 高对比文字 + 低调容器 | Telegram、Slack |
| 品牌表达 | 渐变蓝作为唯一亮色强调 | 克制使用，避免花哨 |
| 交互反馈 | 轻量动效 + 即时响应 | 原生 Material 3 节奏 |

### 5.1.2 设计原则回顾（承接 03 文档）

以下五条设计原则直接指导本方案的每个设计决策：

1. **Provider-Agnostic** → 服务商切换器设计需中立、平等展示所有服务商
2. **System-Native** → 工具调用卡片需呈现系统级操作的可信感
3. **Progressive Complexity** → 默认简洁，高级功能按需展开
4. **Privacy by Default** → 不展示任何暗示云同步的 UI 元素
5. **Extensible by Design** → MCP 工具与内置工具视觉一致

#### 顶栏设计

| 元素 | 规格 | 交互 |
|------|------|------|
| 汉堡菜单 ☰ | 24dp icon，左侧 16dp 内边距 | 点击展开对话列表侧边栏 |

| 服务商切换器 | Chip 样式，显示服务商 icon + 模型名 | 点击展开下拉选择器 |
| 新建对话 | 右侧 + 图标 | 点击创建空白对话 |

#### 服务商切换器（核心交互组件）

切换器是本产品区别于所有竞品的标志性交互。设计目标：**1 次点击完成切换，0 上下文丢失**。

**收起态**：
```
┌──────────────────────────┐
│  [🔵] Claude 3.5 Sonnet ▾│
└──────────────────────────┘
```
- Chip 形态，圆角 20dp
- 左侧：服务商品牌色圆点（8dp），各服务商颜色不同
- 中间：模型名称，单行截断
- 右侧：下拉箭头

**展开态**（底部弹出 BottomSheet）：
```
┌──────────────────────────────────┐
│  选择服务商                  ✕    │
├──────────────────────────────────┤
│  ★ 已配置的服务实例                │
│  ┌────────────────────────────┐  │
│  │ 🔵 Anthropic               │  │
│  │    Claude 3.5 Sonnet    ✓  │  │
│  ├────────────────────────────┤  │
│  │ 🟢 DeepSeek                │  │
│  │    deepseek-chat           │  │
│  ├────────────────────────────┤  │
│  │ 🟣 OpenAI                  │  │
│  │    GPT-4o                  │  │
│  ├────────────────────────────┤  │
│  │ ⚪ Free                    │  │
│  │    免费体验                 │  │
│  └────────────────────────────┘  │
│                                  │
│  [+ 前往设置添加更多服务商]        │
└──────────────────────────────────┘
```

**设计要点**：
- 仅展示已配置且有效的服务实例，避免 18 家全列出造成选择过载
- 当前选中项右侧显示 ✓，背景使用 `primaryContainer` 色
- 每个服务商使用独立品牌色圆点，快速视觉识别
- 底部提供跳转设置的入口，渐进式引导用户添加更多服务商
- 切换后在消息流中插入一条系统消息：「已切换至 [服务商名] ([模型名])」

#### 工具调用卡片

工具调用是 Hyper-Claw 的核心差异化体验。卡片设计需要传达三层信息：**什么工具** → **正在做什么** → **结果是什么**。

**执行中状态**：
```
┌────────────────────────────────────┐
│  🔧 web_search          ● 执行中   │
│  搜索: "Kotlin coroutine deadlock" │
└────────────────────────────────────┘
```

**完成状态（收起）**：
```
┌────────────────────────────────────┐
│  ✅ web_search            0.8s  ▸  │
└────────────────────────────────────┘
```

**完成状态（展开）**：
```
┌────────────────────────────────────┐
│  ✅ web_search            0.8s  ▾  │
├────────────────────────────────────┤
│  找到 5 条结果:                     │
│  1. Stack Overflow: Kotlin...      │
│  2. JetBrains Blog: Structured...  │
│  ...                               │
└────────────────────────────────────┘
```

| 属性 | 规格 |
|------|------|
| 背景色 | `surface`，带 1dp `outlineVariant` 边框 |
| 圆角 | 12dp |
| 内边距 | 12dp |
| 工具名 | 等宽字体，`secondary` 色 |
| 状态指示 | 执行中：`primary` 色脉冲圆点；成功：✅；失败：❌ |
| 耗时显示 | 右侧灰色小字，仅完成后显示 |
| 展开/收起 | 默认收起，点击 ▸ 展开结果详情 |
| 最小显示时长 | 2000ms（防止闪烁） |
| 并行工具 | 垂直堆叠，间距 4dp |

## 5.3 交互设计方案

### 5.3.1 核心交互流程

#### 流程 1: 发送消息与工具调用

```
用户输入文字 → 点击发送
  → 输入框清空，用户消息气泡出现
  → AI 回复区域出现打字指示器（三点跳动）
  → [如有工具调用]
      → 工具卡片出现（执行中状态，脉冲动画）
      → 最小显示 2000ms
      → 执行完成 → 卡片切换为完成态
      → [如有多个工具] 并行显示多张卡片
      → 达到 15 次迭代限制 → 显示提示："已达工具调用上限，正在生成最终回复..."
  → AI 文字逐字/逐块流式渲染
  → 回复完成 → 打字指示器消失
```

#### 流程 2: 服务商切换

```
点击顶部服务商 Chip
  → BottomSheet 从底部弹起（300ms ease-out）
  → 展示已配置服务实例列表
  → 点击目标服务商
      → BottomSheet 收起
      → 顶部 Chip 更新为新服务商
      → 消息流中插入系统消息：
        "── 已切换至 DeepSeek (deepseek-chat) ──"
      → 下一条消息使用新服务商发送
      → 对话上下文完整保持
```

#### 流程 3: 容灾自动切换

```
用户发送消息 → 请求当前服务商
  → API 返回错误（503/429/timeout）
  → 重试（最多 2 次，指数退避）
  → 仍失败 → 消息流中显示黄色警告：
    "⚠️ DeepSeek 暂时不可用，正在切换备用服务商..."
  → 自动尝试下一个可用服务实例
  → 成功 → 消息流中显示通知：
    "✅ 已自动切换至 OpenAI (GPT-4o)"
  → AI 正常回复
  → 全部失败 → 红色错误卡片：
    "❌ 所有服务商暂时不可用，请稍后重试"
```

#### 流程 4: 心跳通知响应

```
[后台] 心跳定时器触发
  → 构建心跳提示词（任务 + 邮件 + 记忆候选）
  → AI 判断是否有需要关注的事项
      → 无事项 → HEARTBEAT_OK，静默
      → 有事项 → 推送系统通知（结构化摘要）
  → 用户点击通知
      → 打开 App，自动定位到心跳对话
      → 展示完整检查报告
      → 用户可直接在对话中下达指令处理
```

## 5.5 动效设计

### 5.5.1 动效原则

- **有意义**：每个动效服务于信息传达或状态转换，拒绝装饰性动效
- **快速**：动效时长 150-300ms，不让用户等待
- **一致**：全局统一使用 Material 3 的 Easing 曲线

### 5.5.2 动效清单

| 场景 | 动效类型 | 时长 | 缓动曲线 | 说明 |
|------|----------|------|----------|------|
| 消息出现 | 淡入 + 上移 | 200ms | EaseOut | 新消息从底部 16dp 位置淡入 |
| 工具卡片出现 | 淡入 + 缩放 | 200ms | EaseOut | 从 95% 缩放到 100% |
| 工具执行中 | 脉冲 | 1500ms 循环 | EaseInOut | 状态圆点透明度 40%↔100% |
| 打字指示器 | 三点依次跳动 | 600ms 循环 | EaseInOut | 3 个圆点依次上移 4dp |
| BottomSheet 弹出 | 底部滑入 | 300ms | EaseOut | 标准 Material BottomSheet |
| 侧边栏展开 | 左侧滑入 + 遮罩淡入 | 250ms | EaseOut | 遮罩最终透明度 40% |
| Chip 选中切换 | 填充色渐变 | 150ms | Linear | 背景色平滑过渡 |
| 错误抖动 | 水平震动 | 300ms | — | 输入验证失败时，输入框左右抖动 2 次 |
| Toast 提示 | 底部弹入 + 自动消失 | 入 200ms / 出 150ms | EaseOut / EaseIn | 2 秒后自动消失 |

### 5.5.3 流式文字渲染

AI 回复采用流式渲染（SSE），文字逐块追加到气泡中。

| 属性 | 规格 |
|------|------|
| 渲染粒度 | 按 SSE data 块追加，通常为 1-3 个 token |
| 光标效果 | 末尾显示竖线光标 `│`（`primary` 色，闪烁 500ms） |
| 自动滚动 | 新内容出现时自动滚动到底部 |
| 滚动打断 | 用户手动上滑时暂停自动滚动，显示"↓ 新消息"浮标 |

## 5.7 无障碍设计

### 5.7.1 原则

- 所有交互元素提供 Content Description
- 色彩对比度满足 WCAG AA（4.5:1 正文，3:1 大字）
- 支持系统字号缩放
- 不依赖纯色彩传达信息（辅以图标/文字）

## 5.9 设计决策记录

| # | 决策 | 原因 | 备选方案 | 否决理由 |
|---|------|------|----------|----------|
| D1 | 服务商切换用 BottomSheet 而非 Dropdown | 手机端 Dropdown 触达区域小，BottomSheet 操作更舒适 | Dropdown 下拉 | 手指够不到顶部列表末尾 |
| D2 | 工具卡片默认收起 | 避免工具结果（最大 8000 字符）撑爆消息流 | 默认展开 | 消息流过长，影响阅读连贯性 |
| D3 | 系统消息居中显示 | 与聊天气泡视觉区分，不占用对话主线注意力 | 左对齐灰色气泡 | 容易与 AI 回复混淆 |
| D4 | 心跳对话独立分组 | 避免自动生成的心跳报告淹没用户手动对话 | 混入时间线 | 用户找不到自己的对话 |
| D5 | Shell 工具首次启用需确认 | 安全考量——Shell 是唯一默认关闭的工具 | 直接 Toggle | 误触风险高，后果严重 |
| D6 | 快捷入口用 2×2 而非列表 | 4 个入口刚好 2×2，视觉平衡且触达面积大 | 水平滚动列表 | 不够直观，需要额外滑动 |
| D7 | 亮/暗模式跟随系统 + 手动覆写 | 尊重系统偏好，同时给予控制权 | 仅跟随系统 | 部分用户有独立偏好 |
