# Hyper-Claw 设计总结

> **文档版本**: v1.0
> **更新日期**: 2025-06-15

---

## 一、Hyper-Claw 概述

Hyper-Claw 是一款面向技术用户的 HyperOS 系统级 AI 助手，基于开源项目 Kai 二次开发，采用 Kotlin Multiplatform + Compose Multiplatform 实现 Android / Desktop / Web 三端覆盖。

### 产品定位

填补移动端 AI 产品的一个显著空白：**没有任何一款产品同时具备「多服务商支持 + 系统级工具调用 + 移动原生体验 + 可扩展工具生态」四重能力。**

| 能力维度 | 竞品现状 | Hyper-Claw 方案 |
|----------|---------|-----------------|
| 多服务商 | ChatGPT / Gemini 均锁定单一服务商 | 18 家服务商，3 种协议族统一适配 |
| 系统级工具 | 第三方 App 无法调用通知、日历、闹钟 | 24 个内置工具，覆盖通知 / 日历 / 闹钟 / Shell |
| 移动原生 | Poe 等聚合器本质是 WebView 套壳 | KMP 原生编译，非 WebView |
| 可扩展性 | 无产品支持 MCP 协议 | MCP (JSON-RPC 2.0) + 8 款预配置服务器 |

### 目标用户

Android 开发者、SRE/DevOps 工程师、独立开发者——日均使用 3+ 款 AI 工具、对隐私敏感、有工具扩展需求的技术群体。

### 核心功能

| 模块 | 说明 |
|------|------|
| 多服务商聊天引擎 | 18 家 AI 服务商一键切换，对话上下文跨服务商保持，含容灾自动故障转移 |
| 工具系统 | 24 个内置工具（信息获取 / 记忆 / 任务调度 / 邮件 / 系统集成）+ MCP 动态扩展 |
| 记忆系统 | 4 类别（通用 / 学习 / 错误 / 偏好），存储 → 强化 → 晋升生命周期 |
| 心跳巡检 | 端侧定时自检，主动聚合任务、邮件、记忆候选并推送通知 |
| 隐私架构 | 对话 XOR 加密、API Key 安全存储、零云同步、无遥测 |

## 二、如何定义需求？

### 2.1 调研方法

需求定义经历了「定量数据 → 定性访谈 → 痛点聚类 → 优先级排序」四步。

**定量层面**：通过 Perplexity AI 聚合 Gartner、Statista 及各厂商财报，获取市场规模（2025 年全球 AI 助手市场 $14.2B）、用户行为数据（开发者日均 3.2 个 AI 工具、71% 对碎片化不满）等关键指标。所有数据经人工交叉验证。

**定性层面**：对 15 位目标用户进行 45-60 分钟深度访谈（深圳 / 北京 / 杭州 / 远程），覆盖 Android 开发者 6 人、SRE/DevOps 5 人、独立开发者 4 人。访谈围绕 AI 使用习惯、痛点、隐私态度、付费意愿展开。

### 2.2 痛点提取

访谈录音转写后，使用 Claude 进行主题编码（thematic coding），提取 7 个一级痛点维度：

| 痛点 | 提及率 | 核心描述 |
|------|--------|----------|
| 上下文碎片化 | 87% | 切换 AI 工具时上下文丢失 |
| 工具隔离 | 73% | AI 只能聊天，无法操作系统功能 |
| 隐私顾虑 | 67% | 敏感代码不能发给云端 AI |
| 冷启动 | 60% | AI 不记得用户偏好和历史 |
| 供应商锁定 | 53% | 依赖单一服务商，宕机无替代 |
| 被动交互 | 47% | AI 不主动检查任务和邮件 |
| 扩展性不足 | 40% | 无法添加自定义工具 |

### 2.3 需求优先级

使用 ChatGPT 模拟 Kano 模型分析需求类型（基本型 / 期望型 / 兴奋型），映射为 MoSCoW 框架：

- **Must Have**：多服务商引擎、内置工具系统、对话加密、多语言
- **Should Have**：MCP 集成、心跳巡检、邮件管理、记忆系统
- **Could Have**：Shell 执行、容灾链路、桌面端
- **Won't Have（本版本）**：语音交互、图像生成、多用户协作

每个痛点至少被 1 项需求覆盖，验证了需求集的完整性。

### 2.4 AI 工具参与

| 环节 | 工具 | 具体应用 |
|------|------|----------|
| 市场数据采集 | Perplexity AI | 多源聚合 Gartner / Statista 数据 |
| 痛点主题编码 | Claude | 从 15 人访谈中提取 7 个维度 + 统计频率 |
| 需求优先级排序 | ChatGPT | Kano 模型模拟 → MoSCoW 映射 |
| 性能指标制定 | Claude | 参考 Google HEART 框架设定阈值 |

## 三、如何进行设计调研？

### 3.1 竞品分析

从 12 款候选产品中，基于用户覆盖度、功能深度、技术路线差异性，筛选出 6 款核心竞品深度分析：ChatGPT App、Google Gemini、Samsung Galaxy AI、Apple Intelligence、Poe、Open Interpreter。

**关键结论**：市场存在明显空白——尚无产品同时具备多服务商 + 系统级工具 + 移动原生 + 可扩展工具生态。每款竞品至少在 2 个维度上存在硬伤。

### 3.2 用户画像

基于访谈数据构建 3 个 Persona，覆盖核心用户群体：

| Persona | 职业 | 核心痛点 | 映射功能 |
|---------|------|----------|----------|
| 张明 27 岁 | Android 开发 | 日均切换 3 个 AI，上下文断裂 | 多服务商引擎、记忆系统、容灾链路 |
| 李雪 32 岁 | SRE/DevOps | AI 被动，不主动检查邮件任务 | 心跳巡检、邮件工具链、通知推送 |
| David 35 岁 | 独立开发者 | 客户代码不能发云端，无法扩展工具 | Ollama 本地模型、MCP 自定义服务器 |

### 3.3 场景验证

通过 6 个核心使用场景验证功能设计的合理性：

1. **多服务商编程助手**（张明）— 聊天内切换服务商 + 容灾自动故障转移
2. **主动晨间简报**（李雪）— 心跳触发 → 信息聚合 → 通知推送
3. **通勤邮件分诊**（李雪）— 邮件 6 工具链路 + AI 优先级评分
4. **MCP 自定义工作流**（David）— 预配置服务器 + 自定义 MCP 端点
5. **渐进式偏好学习**（张明）— 记忆存储 → 强化 → 晋升完整生命周期
6. **定时任务 + Shell**（David）— cron 调度 + 安全沙箱执行

每个场景以故事板形式描述完整用户操作路径，并绘制了「主动晨间简报」的用户旅程地图（触点 / 情绪 / 机会点）。最终通过痛点→方案映射表确认 7 个痛点全覆盖。

| 画像生成 | Claude + Midjourney | 行为特征生成 + 概念照片 |

## 四、如何进行方案设计？

### 4.1 设计原则

从竞品弱点和用户痛点中提炼 5 条核心原则，指导全部设计决策：

| 原则 | 含义 | 来源痛点 |
|------|------|----------|
| Provider-Agnostic | 不绑定单一 AI 服务商 | P5 供应商锁定 |
| System-Native | 深度集成 OS 能力，非 WebView | P2 工具隔离 |
| Progressive Complexity | 新手开箱即用，专家按需深入 | 功能复杂度管理 |
| Privacy by Default | 全部数据本地存储、加密、零云同步 | P3 隐私顾虑 |
| Extensible by Design | MCP 开放协议实现工具生态扩展 | P7 扩展性不足 |

### 4.2 架构选型

每个核心模块经历了 2-3 个方案比选，选定最优方案：

| 模块 | 选定方案 | 否决方案 | 否决原因 |
|------|----------|----------|----------|
| 服务商架构 | 多服务商统一 API 层 | 单服务商深度集成 / 纯聚合器 | 违反 Provider-Agnostic / 引入第三方隐私风险 |
| 工具系统 | 内置工具 + MCP 动态扩展 | 纯硬编码 / 纯插件 | 扩展性差 / 冷启动体验差 |
| 记忆系统 | 分类记忆 + 强化生命周期 | 简单 KV / 向量数据库 | 无学习信号 / 端侧资源消耗过大 |
| 主动 AI | 端侧定时心跳 | 服务器推送 / 事件驱动 | 违反隐私原则 / 电池消耗不可控 |
| 导航结构 | 2-Screen（聊天 + 设置） | 3-Tab（Chat/Tools/Settings） | Tools Tab 利用率极低，违反 AI-first |

### 4.3 界面设计方案

基于 Material 3 设计系统，品牌色 `#3366FF`（亮色）/ `#6B8FFF`（暗色）。

**聊天主界面**（95% 使用时间）：
- 顶栏集成服务商切换器（Chip 形态，点击弹出 BottomSheet 选择）
- 消息流：用户气泡右对齐（`primaryContainer`）、AI 气泡左对齐（`surfaceVariant`）、系统消息居中
- 工具调用以折叠卡片呈现（执行中脉冲动画 → 完成后收起，点击展开结果）
- 输入框渐变蓝边框（品牌标识），多行自适应
- 左侧抽屉放置对话列表（历史对话按日期分组、心跳记录独立分组）

**设置页**：分组列表布局，5 个 Section（AI 服务商 / 工具 / 智能功能 / 个性化 / 数据与隐私），二级页面处理具体配置（API Key 安全输入、工具独立 Toggle、MCP 服务器管理）。

**响应式策略**：手机 < 600dp 单栏 + Drawer；平板/桌面 > 840dp 双栏（左侧 280dp 对话列表 + 右侧聊天区）。

### 4.4 交互设计要点

| 交互 | 方案 | 设计理由 |
|------|------|----------|
| 服务商切换 | 聊天顶栏一键切换，非设置页 | 日均切换 5+ 次，深层路径成本过高 |
| 工具调用展示 | 默认收起，点击展开 | 工具结果最大 8000 字符，展开会撑爆消息流 |
| 容灾通知 | 消息流内黄色警告 + 自动切换 | 不打断用户，自动恢复 |
| Shell 启用 | 首次开启需安全确认弹窗 | 唯一默认关闭的工具，误触后果严重 |
| 心跳报告 | 系统通知 → 点击进入心跳对话 | 不打扰主对话，独立查看 |

### 4.5 AI 工具参与

| 环节 | 工具 | 具体应用 |
|------|------|----------|
| 设计原则提炼 | Claude | 从竞品分析和痛点归纳 5 条原则 |
| 架构方案评估 | Claude | 可维护性 / 用户价值 / 隐私三维度打分 |
| 记忆方案比选 | ChatGPT | 存储成本 / 检索精度 / 端侧性能三维度对比 |
| 心跳模式发散 | ChatGPT | 头脑风暴 8 种模式 → 筛选至 3 种 |
| 交互线框图 | Figma AI | A/B 两版导航方案，5 人可用性测试选定 B |
| 色彩系统推导 | Figma AI | 基于 `#3366FF` 推导 Material 3 完整色槽 |
| 安全边界设计 | Claude | 参考 OWASP LLM Top 10 制定工具安全策略 |
