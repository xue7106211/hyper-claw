#### 3 种协议族适配

**OpenAI-Compatible**（覆盖 15/18 服务商）：
- 端点格式：`{baseUrl}/chat/completions`
- 工具调用：标准 `tool_calls` 字段
- 流式响应：SSE `data:` 行

**Gemini**（Google）：
- 端点格式：`/v1beta/models/{model}:generateContent`
- 消息格式：`contents[].parts[]` 结构
- 工具调用：`functionCall` / `functionResponse` 格式

**Anthropic**（Claude）：
- 端点格式：`/v1/messages`
- 消息格式：`content[]` blocks，支持 `tool_use` / `tool_result` 类型
- 模型列表：`/v1/models`

#### 容灾链路

```
用户请求 → 获取有序服务实例列表
  → 实例 1: 请求
       → 成功 → 返回
       → 失败 → 重试（最多 2 次，指数退避）
              → 仍失败 → 判断是否可重试错误
                   → 配额耗尽等不可重试 → 跳过重试
                   → 可重试 → 继续退避
       → 所有重试失败 → 故障转移
  → 实例 2: 同上流程
  → ...
  → 实例 N: 同上流程
  → [可选] Free 服务商作为最终兜底
  → 全部失败 → 向用户报错
```

#### 工具调用循环机制

| 参数 | 值 | 说明 |
|------|-----|------|
| MAX_TOOL_ITERATIONS | 15 | 单次用户请求最多 15 轮工具调用 |
| MAX_REPEATED_TOOL_CALLS | 3 | 连续 3 次相同调用批次触发终止 |
| MAX_API_RETRIES | 2 | 每次 API 调用最多重试 2 次 |
| MAX_TOOL_RESULT_LENGTH | 8,000 字符 | 单次工具结果截断阈值 |
| MIN_TOOL_DISPLAY_MS | 2,000ms | 工具执行卡片最小显示时长 |

**并行执行**：同一轮中的多个工具调用通过 `coroutineScope { async {} }.awaitAll()` 并行执行。

**优雅降级**：达到 15 次迭代限制后，发起一次**不带工具**的最终 API 调用，系统提示："You have reached the tool call limit. Please respond with the best answer you have so far."

### 4.2.3 MCP 集成

#### 协议规格

| 项目 | 规格 |
|------|------|
| 协议 | JSON-RPC 2.0 over HTTP POST |
| 协议版本 | 2024-11-05 |
| 响应格式 | JSON 或 SSE (text/event-stream) |
| 会话追踪 | Mcp-Session-Id Header |
| 请求超时 | 30 秒 |
| 连接超时 | 10 秒 |
| 客户端标识 | name: "Hyper-Claw", version: "1.0" |

#### 连接流程

```
1. initialize(protocolVersion, clientInfo)
   ← 服务器返回 capabilities
2. notifications/initialized（通知，无响应）
3. tools/list
   ← 服务器返回工具定义列表
4. 注册为本地 Tool 对象（McpTool 适配器）
   - Tool ID: mcp_{serverId}_{toolName}
   - 超时: 60 秒
5. 就绪，工具可被 AI 调用
```

### 4.2.4 心跳/主动 AI

#### 配置参数

| 参数 | 字段 | 默认值 | 约束 |
|------|------|--------|------|
| 启用状态 | enabled | true | Boolean |
| 检查间隔 | intervalMinutes | 30 | ≥ 5 分钟 |
| 活跃开始 | activeHoursStart | 8 | 0-23 |
| 活跃结束 | activeHoursEnd | 22 | 0-23 |
| 自定义提示词 | customPrompt | (默认) | 自由文本 |

#### 心跳检查聚合内容

1. **待处理任务**：从 TaskStore 获取 PENDING 状态任务，包含：
   - 任务描述
   - 任务 ID
   - 调度时间（ISO 8601 格式）
   - cron 表达式（如有）

2. **邮件状态**：从 EmailStore 获取各账户状态：
   - 账户邮箱地址
   - 未读邮件数量
   - 最后同步时间

3. **记忆晋升候选**：从 MemoryStore 获取 hitCount ≥ 5 的记忆条目：
   - 记忆键名
   - 命中次数
   - 类别
   - 内容摘要

#### 心跳日志

保留最近 5 条执行记录，每条包含：
- 时间戳（epoch 毫秒）
- 执行结果（成功/失败）

### 4.2.6 记忆系统

#### 4 个记忆类别

| 类别 | 枚举值 | 用途 | 对应工具 |
|------|--------|------|----------|
| 通用记忆 | GENERAL | 用户姓名、项目细节等事实性记忆 | memory_store |
| 学习经验 | LEARNING | 有效的代码模式、成功方案 | memory_learn |
| 错误解决 | ERROR | 调试经验、错误修复方案 | memory_learn |
| 用户偏好 | PREFERENCE | 编码风格、输出格式偏好 | memory_learn |

#### 记忆条目结构

```
MemoryEntry {
  key: String          // 描述性键名（如 "code_indent_preference"）
  content: String      // 记忆内容
  category: Enum       // GENERAL | LEARNING | ERROR | PREFERENCE
  hitCount: Int        // 命中计数（默认 1）
  source: String?      // 来源：user_correction / observation / error_resolution
  createdAt: Long      // 创建时间（epoch ms）
  updatedAt: Long      // 最后更新时间（epoch ms）
}
```

#### 生命周期流转

| 阶段 | 触发条件 | 操作 | 结果 |
|------|----------|------|------|
| **存储** | 用户纠正、AI 观察、错误解决 | memory_store 或 memory_learn | 新建条目，hitCount=1 |
| **强化** | 记忆被成功应用 | memory_reinforce | hitCount++ |
| **候选** | hitCount ≥ 5 | 心跳检查时自动呈现 | 进入晋升候选列表 |
| **晋升** | AI 或用户确认晋升 | promote_learning | 内容写入 Soul，原记忆删除 |
| **遗忘** | 用户主动删除 | memory_forget | 条目被移除 |

## 4.4 数据架构与安全

### 4.4.1 对话加密存储

| 项目 | 规格 |
|------|------|
| 加密算法 | XOR 对称加密 |
| 密钥长度 | 32 字节随机生成 |
| 密钥编码 | Base64 |
| 密钥存储位置 | AppSettings（平台安全存储包裹） |
| Android 密钥保护 | EncryptedSharedPreferences + AES256_GCM MasterKey |
| 加密文件名 | conversations.enc |
| 数据格式 | JSON 序列化 (ConversationsData, version=2) |

**加密流程**：
```
对话数据 → JSON 序列化 → UTF-8 字节 → XOR(bytes, key[i % keyLen]) → 加密字节 → 写入文件
```

**解密流程**：
```
读取文件 → 加密字节 → XOR(bytes, key[i % keyLen]) → UTF-8 字节 → JSON 反序列化 → 对话数据
```

### 4.4.2 隐私架构

| 维度 | 实现 |
|------|------|
| 数据存储 | 全部本地，零云同步 |
| 遥测收集 | 无，不收集任何用户行为数据 |
| API Key | 仅存于设备安全存储 |
| 邮件密码 | 独立加密存储（per-account） |
| 数据可移植 | JSON 格式导出/导入 |
| 导入分区 | 可选择性导入：SERVICES / SOUL / MEMORY / SCHEDULING / HEARTBEAT / EMAIL / TOOLS / MCP |

## 4.6 AI 工具使用全流程总结

以下汇总表记录了整个产品设计过程中 AI 工具的使用情况，体现了人机协同的设计方法论。

| 阶段 | AI 工具 | 用途 | 产出 |
|------|---------|------|------|
| 市场调研 | Perplexity AI | 聚合市场报告和行业数据 | 市场规模数据、用户统计 |
| 竞品分析 | Claude | 分析 12 款产品功能矩阵，提取差异化维度 | 6 款核心竞品对比表 |
| 用户研究 | Claude | 访谈录音转写 + 主题编码 (thematic coding) | 7 个痛点维度、提及频率统计 |
| 需求优先级 | ChatGPT | 模拟 Kano 模型分析需求类型 | MoSCoW 优先级分类 |
| 用户画像 | Claude + Midjourney | 生成初始画像 + 概念插图描述 | 3 个 Persona 卡片 |
| 场景设计 | Claude | 头脑风暴 20+ 候选场景并筛选 | 6 个核心使用场景 |
| 故事板 | Midjourney | 为每个场景生成概念插图描述 | 场景视觉化辅助 |
| 旅程地图 | ChatGPT | 生成旅程地图框架模板 | 晨间简报旅程图 |
| 设计原则 | Claude | 从竞品分析和痛点提炼原则 | 5 条核心设计原则 |
| 架构探索 | Claude | 评估 3 种架构方案的 trade-off | 多服务商统一 API 层方案 |
| 记忆方案 | ChatGPT | 对比 3 种方案的可行性（存储/精度/性能） | 分类记忆 + 强化生命周期方案 |
| 心跳方案 | ChatGPT | 头脑风暴 8 种主动 AI 模式并评估 | 端侧定时心跳方案 |
| 交互设计 | Figma AI | 生成 A/B 两版线框图 | 2-Screen 导航 + 聊天内切换 |
| 设计系统 | Figma AI | 基于主色自动生成 Material 3 色槽 | 完整暗色/亮色色彩体系 |
| 非功能需求 | Claude | 参考 HEART 框架制定性能阈值 | 性能指标定义 |
| 安全设计 | Claude | 探索 MCP 工具安全最佳实践 | 工具安全边界（黑名单/截断/限速） |

**工具使用覆盖率**：
- **Perplexity AI**：市场数据聚合（1 个阶段）
- **Claude**：深度分析和评估（8 个阶段）
- **ChatGPT**：创意发散和模型模拟（4 个阶段）
- **Midjourney**：视觉化辅助（2 个阶段）
- **Figma AI**：交互和视觉设计（2 个阶段）

全流程体现了「AI 辅助决策、人工把关质量」的协同模式。每个 AI 产出都经过人工审核、校准和整合，确保设计质量和一致性。
