## 3.1 设计原则

基于竞品分析（1.2 节）和用户痛点（1.3 节），我们提炼出 5 条核心设计原则，作为后续所有设计决策的评判标准。

> 🤖 **AI 工具标注**: 设计原则由 Claude 基于竞品分析和用户痛点提炼，经团队设计评审后确定 5 条核心原则。

### 原则 1: Provider-Agnostic（服务商无关）

**定义**：不绑定任何单一 AI 服务商，用户自由选择和切换。

**理由**：痛点 P5（供应商锁定，53%）表明超过一半的目标用户对单一供应商依赖感到不安。AI 行业格局变化极快，今天最好的模型明天可能被超越。产品不应押注任何一家。

**落地要求**：
- 统一 API 适配层，屏蔽 3 种协议族差异（OpenAI-Compatible、Gemini、Anthropic）
- 每个服务商独立配置（API Key、模型、端点），互不干扰
- 支持 OpenAI-Compatible 自定义端点，兼容未来任何兼容服务商

### 原则 2: System-Native（系统原生）

**定义**：深度集成操作系统能力，而非 WebView 套壳。

**理由**：痛点 P2（工具隔离，73%）的核心在于——AI 只能聊天，不能「做事」。而真正的「做事」需要操作系统权限：发通知、设闹钟、读日历、执行命令。这是任何 Web 应用无法企及的。

**落地要求**：
- 原生访问 Android 通知系统（NotificationManager）
- 原生访问日历（CalendarContract）和闹钟（AlarmClock Intent）
- Shell 命令执行（Runtime.exec / ProcessBuilder）
- 前台服务保活心跳调度

### 原则 3: Progressive Complexity（渐进式复杂度）

**定义**：新手开箱即用，专家按需深入。

**理由**：18 家服务商、20+ 工具、MCP 协议——功能强大但也意味着复杂。如果上手门槛过高，会流失大量用户。

**落地要求**：
- 提供 Free 服务商，零配置即可体验核心对话能力
- 高级功能（MCP、心跳、Shell）藏在设置深层，有默认值
- 工具默认启用合理子集，高级工具（如 Shell）默认关闭

### 原则 4: Privacy by Default（默认隐私）

**定义**：所有数据本地存储、加密处理、零云同步。

**理由**：痛点 P3（隐私顾虑，67%）在开发者群体中尤为突出。独立开发者有 NDA 约束，企业开发者有合规要求。隐私不是可选项，而是默认状态。

**落地要求**：
- 对话 XOR 加密存储，密钥存于平台安全存储（Android EncryptedSharedPreferences/AES256_GCM）
- API Key 仅本地保存
- 无用户行为遥测
- 设置导出/导入支持数据可移植

### 原则 5: Extensible by Design（可扩展设计）

**定义**：通过开放协议实现工具生态无限扩展。

**理由**：痛点 P7（扩展性不足，40%）表明用户需要超越内置工具的能力。但无限制的插件系统会带来安全风险和质量问题。MCP 协议提供了结构化的扩展路径。

**落地要求**：
- MCP 协议（JSON-RPC 2.0）作为标准扩展接口
- 动态工具发现（tools/list），无需硬编码
- 预配置热门服务器降低上手成本
- 每个 MCP 服务器独立开关，用户完全掌控

## 3.2 架构方案探索

### 方案 A：单服务商深度集成

**描述**：选择一家 AI 服务商（如 OpenAI）进行深度集成，充分利用其全部 API 能力。

**优势**：
- 实现最简单，只需维护一套 API 适配
- 可深度利用服务商特有功能（如 OpenAI 的 Assistants API、Code Interpreter）
- 文档和社区支持最丰富

**劣势**：
- 用户完全锁定在单一供应商
- 服务商宕机 = 产品不可用
- 无法利用不同模型的各自优势（DeepSeek 快且廉价、Claude 推理强）
- 与设计原则 1（Provider-Agnostic）根本冲突

**风险评估**：服务商政策变更（涨价、限速、停服）将直接威胁产品存亡。2024 年多次 AI 服务商宕机事件证明了这一风险的现实性。

**结论**：❌ 否决 — 与 Provider-Agnostic 原则根本冲突，单点故障风险不可接受。

---

### 方案 B：多服务商统一 API 层 ✅

**描述**：构建统一的 API 适配层，将 3 种协议族的差异封装在适配器内部，对上层暴露统一接口。

**3 种协议族**：

| 协议族 | 覆盖服务商 | API 特点 |
|--------|-----------|----------|
| OpenAI-Compatible | OpenAI, DeepSeek, Mistral, Groq, xAI, OpenRouter, NVIDIA, Cerebras, Ollama Cloud, LongCat, Together AI, Hugging Face, Venice AI, Moonshot AI, OpenAI-Compatible 自定义端点 | `/v1/chat/completions`，标准 tool_calls 格式 |
| Gemini | Google Gemini | `/v1beta/models/{model}`，独特的 content parts 格式 |
| Anthropic | Anthropic (Claude) | `/v1/messages`，content blocks + tool_use 类型 |

**优势**：
- 用户自由选择 18 家服务商，按需切换
- 容灾链路：服务商 A 宕机 → 自动故障转移到服务商 B
- 面向未来：大部分新兴 AI 服务商都采用 OpenAI-Compatible 格式，零成本接入
- 多实例支持：同一服务商可配置多个实例（不同 API Key / 模型）

**劣势**：
- 需要维护 3 套协议适配器
- 不同服务商的工具调用格式有差异，需统一处理

**缓解措施**：
- 15/18 家服务商使用 OpenAI-Compatible 格式，实际只需 2 个额外适配器（Gemini + Anthropic）
- 统一的工具调用抽象层封装格式差异

**结论**：✅ 选中 — 最佳平衡点。18 家服务商统一接入，3 种协议族适配，兼顾用户自由与工程可维护性。

> 🤖 **AI 工具标注**: 三种架构方案由 Claude 评估各自的 trade-off，结合可维护性、用户价值、隐私保护三个维度打分。方案 B 综合得分最高。

---

### 方案 C：纯聚合器（OpenRouter 模式）

**描述**：将所有请求路由到一个聚合服务（如 OpenRouter），由聚合方负责多模型分发。

**优势**：
- 集成最简单，只需对接一个 API
- 一个 API Key 访问所有模型

**劣势**：
- 引入第三方依赖，聚合器宕机 = 产品不可用
- 额外延迟（请求多经过一层）
- 成本加价（聚合方收取服务费）
- 所有请求经过第三方，隐私风险增大
- 无法支持本地模型（Ollama）

**结论**：❌ 否决 — 违反 Privacy by Default 和自主可控原则。引入不必要的第三方依赖。

## 3.3 工具系统设计探索

### 方案 1：纯硬编码工具

**描述**：所有工具都内置在 App 二进制文件中，不提供任何扩展机制。

**优势**：完全控制工具质量、安全性、性能。

**劣势**：每添加一个新工具都需要发版更新。用户无法根据自己的需求扩展。与设计原则 5（Extensible by Design）冲突。

**结论**：❌ 否决 — 扩展性差，无法满足 P7 痛点。

---

### 方案 2：内置工具 + MCP 动态工具 ✅

**描述**：核心工具硬编码在 App 中（保证质量和安全），同时通过 MCP 协议支持动态工具扩展。

**内置工具分类（20+ 工具，5 大类）**：

| 类别 | 工具 | 数量 |
|------|------|------|
| 信息获取 | web_search, get_local_time, get_location_from_ip, open_url | 4 |
| 记忆管理 | memory_store, memory_forget, memory_learn, memory_reinforce | 4 |
| 任务调度 | schedule_task, cancel_task, list_tasks | 3 |
| 心跳系统 | configure_heartbeat, trigger_heartbeat, promote_learning | 3 |
| 邮件管理 | setup_email, check_email, read_email, reply_email, compose_email, search_email | 6 |
| 系统集成（平台相关） | send_notification, create_calendar_event, set_alarm, shell_command | 4 |
| **总计** | | **24** |

**MCP 动态扩展**：
- 协议：JSON-RPC 2.0 over HTTP POST + SSE 响应
- 工具发现：`initialize()` → `tools/list` → 自动注册
- 8 款预配置热门服务器开箱可用

**安全边界设计**：

| 安全措施 | 具体实现 |
|----------|----------|
| Shell 命令黑名单 | 14 条正则模式（rm -rf /、恶意递归、shutdown、reboot 等） |
| 工具结果截断 | 单次结果最大 8,000 字符 |
| 执行超时 | 默认 30s，最大 Android 60s / Desktop 120s |
| 工具循环限制 | 单次请求最多 15 次工具迭代 |
| 重复检测 | 连续 3 次相同工具调用批次自动终止 |
| 独立开关 | 每个工具可单独启用/禁用 |
| Shell 默认关闭 | shell_command 默认 `isEnabled = false` |

**结论**：✅ 选中 — 兼顾稳定性与扩展性。内置工具保证核心体验，MCP 协议释放无限可能。

> 🤖 **AI 工具标注**: 工具安全边界由 Claude 探索 MCP 协议安全最佳实践后制定，参考 OWASP LLM Top 10 中 Tool Use 相关条目。

---

### 方案 3：纯插件系统

**描述**：不提供内置工具，所有工具以插件形式按需安装。

**优势**：最大灵活性，App 体积最小。

**劣势**：冷启动体验极差（新用户看到空壳）。第三方插件质量和安全性不可控。核心功能（如记忆系统）依赖第三方不可靠。

**结论**：❌ 否决 — 用户体验和安全性不可控。

---

## 3.4 记忆系统设计探索

### 方案 1：简单 Key-Value 存储

**描述**：记忆以 key→value 平坦结构存储，无分类无权重。

**优势**：实现最简单。

**劣势**：
- 无学习信号——所有记忆权重相同，无法区分重要性
- 无进化路径——存了就存了，不会随使用成长
- 长期积累后信息过载，无法筛选真正有价值的记忆

**结论**：❌ 否决 — 无法实现渐进式学习，与 P4 痛点的深层需求不匹配。

---

### 方案 2：向量数据库

**描述**：使用 Embedding 模型将记忆向量化，存储在端侧向量数据库中，通过语义相似度检索。

**优势**：
- 语义检索能力强，"类似问题"可以自动关联
- 最接近人类记忆的工作方式

**劣势**：
- 需要在端侧运行 Embedding 模型（约 50-200MB），显著增加 App 体积
- 向量索引占用大量存储空间
- 每次检索需要计算余弦相似度，电池消耗不可忽视
- 对于 KMP 跨平台架构，端侧向量数据库的平台兼容性是挑战

**结论**：❌ 否决 — 端侧资源消耗过大，技术复杂度与收益不匹配。

> 🤖 **AI 工具标注**: 三种记忆方案由 ChatGPT 对比可行性，从存储成本、检索精度、端侧性能三个维度评估。方案 3 在端侧场景下综合最优。

---

### 方案 3：分类记忆 + 强化生命周期 ✅

**描述**：将记忆分为 4 个类别，通过"强化计数"（hitCount）实现自然选择——频繁有用的记忆存活并晋升，无用的记忆自然沉寂。

**4 个记忆类别**：

| 类别 | 用途 | 存储方式 |
|------|------|----------|
| GENERAL | 通用事实（用户姓名、项目信息） | memory_store |
| LEARNING | 有效经验（成功的代码模式） | memory_learn |
| ERROR | 错误解决方案（调试经验） | memory_learn |
| PREFERENCE | 用户偏好（编码风格、输出格式） | memory_learn |

**记忆条目数据结构**：
- `key`: 描述性键名
- `content`: 记忆内容
- `category`: 分类（GENERAL / LEARNING / ERROR / PREFERENCE）
- `hitCount`: 命中计数（初始 1，每次强化 +1）
- `source`: 来源（user_correction / observation / error_resolution）
- `createdAt` / `updatedAt`: 时间戳

**生命周期：存储 → 强化 → 晋升**

```
[存储] memory_store / memory_learn
   ↓ 记忆被成功应用
[强化] memory_reinforce (hitCount++)
   ↓ hitCount ≥ 5
[候选] 心跳检查时呈现为"晋升候选"
   ↓ AI 或用户确认
[晋升] promote_learning → 写入系统提示词 (Soul)
   ↓ 原记忆删除（已固化为人格）
[人格] 成为 AI 的永久行为准则
```

**结论**：✅ 选中 — 端侧最优解。轻量级存储，有意义的学习信号，清晰的进化路径，对用户完全透明。

## 3.5 主动 AI（心跳）设计探索

### 方案 1：服务器推送

**描述**：云端服务器定时向设备推送"检查指令"，触发 AI 自检。

**优势**：电池效率最高（利用系统推送通道），投递可靠。

**劣势**：
- 需要搭建和维护云端推送基础设施
- 服务器需要知道"什么时候推送什么"——意味着用户的任务/邮件信息需要上传云端
- 严重违反 Privacy by Default 原则
- 引入云端依赖，与本地优先架构矛盾

**结论**：❌ 否决 — 违反隐私原则。云端推送意味着泄露用户行为模式。

---

### 方案 2：端侧定时心跳 ✅

**描述**：设备本地运行定时器，按配置间隔触发 AI 自检。

**配置参数**：

| 参数 | 默认值 | 范围 |
|------|--------|------|
| enabled | true | 开/关 |
| intervalMinutes | 30 | 最小 5 分钟 |
| activeHoursStart | 8 | 0-23 |
| activeHoursEnd | 22 | 0-23 |
| customPrompt | (默认提示词) | 自由文本 |

**心跳触发流程**：

```
isHeartbeatDue() 检查：
  1. 是否启用？ → 否则跳过
  2. 当前时间是否在活跃时段内？ → 否则跳过
  3. 距上次心跳是否已过间隔？ → 否则跳过
  → 全部通过 → 构建心跳提示词
```

**心跳提示词包含 3 类信息**：
1. **待处理任务**：从 TaskStore 获取，包含描述、ID、调度时间、cron 表达式
2. **邮件状态**：从 EmailStore 获取各账户未读数和最后同步时间
3. **记忆晋升候选**：hitCount ≥ 5 的记忆条目

**默认提示词**：
> "[HEARTBEAT] This is an automatic self-check. Review your memories and pending tasks. If everything looks good and nothing needs attention, respond with exactly: HEARTBEAT_OK. If something needs attention (stale memories, due tasks, user follow-ups), address it."

**Android 实现**：前台服务（Foreground Service）保活，确保后台可靠调度。

**结论**：✅ 选中 — 隐私与主动性的最佳平衡。完全本地运行，用户完全掌控配置。

> 🤖 **AI 工具标注**: 主动 AI 交互模式由 ChatGPT 头脑风暴 8 种模式（定时轮询、事件驱动、服务器推送、混合模式、用户触发、地理围栏、情境感知、对话空闲检测），经可行性和隐私性评估筛选至 3 种候选方案。

---

### 方案 3：事件驱动触发

**描述**：监听系统事件（新邮件到达、日历提醒触发等），在事件发生时触发 AI 处理。

**优势**：最敏捷——事件发生即响应，无延迟。

**劣势**：
- 需要持续监听多种系统事件，Android 后台限制严格（Doze 模式、App Standby）
- 电池消耗不可预估，取决于事件频率
- 不同 Android 版本/厂商的后台策略差异大，兼容性噩梦

**结论**：❌ 否决 — 系统限制多，电池消耗不可控，跨设备一致性差。

## 3.6 交互流设计探索

### 3.6.1 导航结构

**方案 A: 3-Tab 导航（Chat / Tools / Settings）**

传统的三栏底部导航，功能分区清晰。

- Chat Tab：对话界面
- Tools Tab：工具列表和手动触发
- Settings Tab：设置和配置

**分析**：数据显示 Chat 占用户使用时间的 95% 以上。独立的 Tools Tab 利用率极低——因为工具的正确触发方式是 AI 在对话中根据上下文自动调用，而非用户手动点击。

**结论**：❌ Tools Tab 违反 "AI-first" 交互理念

---

**方案 B: 2-Screen 导航（Chat + Settings）✅**

极简双屏结构：

- **聊天主界面**（95% 时间）：对话列表 + 消息流 + 服务切换器 + 输入框
- **设置页**（5% 时间）：服务商管理、工具开关、MCP 配置、记忆管理、心跳设置、邮件账户、安全与隐私

**设计理念**：工具是 AI 的能力，不是独立入口。用户说"帮我查邮件"，AI 自动调用邮件工具——用户不需要知道背后的工具名称和参数。

**结论**：✅ 选中 — 符合 AI-first 交互范式，最大化聊天界面的沉浸感。

> 🤖 **AI 工具标注**: 交互方案由 Figma AI 生成 A/B 两版线框图，经 5 人可用性测试后选定方案 B。测试结果显示 B 方案的任务完成率（92%）显著高于 A 方案（78%）。

---

### 3.6.2 服务切换器位置

**方案 A: 设置页切换**

服务商选择放在设置页面中，用户需要离开聊天界面去切换。

**问题**：场景 1 中张明日均切换 5+ 次，设置页路径太深（聊天 → 设置 → 服务商 → 切换），操作成本过高。

**结论**：❌ 切换频率高的操作不应藏在深层设置

---

**方案 B: 聊天内切换 ✅**

服务商选择器直接集成在聊天界面（顶部或输入框附近），支持快速切换。

- 显示当前服务商和模型名称
- 点击展开下拉列表，显示已配置的服务实例
- 多实例支持：同一服务商的不同配置一目了然

**结论**：✅ 选中 — 一键切换，不打断对话流。

---

### 3.6.3 输入框设计

经过多轮迭代，最终确定：

- **渐变边框**：品牌色 `#3366FF` (brandBlue) → `#6B8FFF` (lightBlue) 渐变，强化品牌识别
- **Placeholder 颜色**：40% 透明度，柔和淡色引导
- **多行支持**：自动扩展高度，适应长文本输入
- **发送按钮**：与品牌主色一致，悬浮于输入框右侧
